# v0.2 Migration Notes Per System

Riso has been fully re-authored as `riso-spec-v0.2.yaml` demonstrating all v0.2 features. The other 6 systems need smaller, targeted updates. This document describes exactly what each system needs.

For each system, the migrations are listed as **required** (rule violations or missing capabilities under v0.2) or **optional** (would benefit from new features but works without them).

The migration changes are small enough that they should be applied directly to each spec.yaml rather than authored as full rewrites. Most are 1-3 line additions per rulebook entry.

---

## Cross-cutting: `check_scope` on every rule (Change 12)

Added after Swiss library work surfaced false-positive failures in batch component checks. Every rulebook entry should now declare:

```yaml
check_scope: artifact | component | both
applies_when: "..."   # optional, for component-scoped rules that need a guard
```

Defaults: omitted `check_scope` is treated as `both`, preserving v0.1 behavior. Authors should annotate explicitly. Most structural rules (zero-radius, no-shadows, palette, type-scale, single-typeface) are `both`. Anything that counts "X per screen" (red-finite-resource, four-color-process) is `artifact`. Anything that vacuously passes when its target element class is absent (metadata-uppercase, zero-padded-indices) is `component` with an `applies_when` predicate.

Apply this migration alongside the per-system migrations below. ~10 extra minutes per system.

## Wireframe Library

### Required

**Add `check_method` to every rulebook entry.** Already mostly there — the wireframe-spec.yaml's rulebook documented `check_method: mechanical | semantic` informally in prose. Make it a structured field per Change 9.

```yaml
rulebook:
  - id: wireframe-mobile-first
    rule: "..."
    severity: required
    check_method: semantic   # ADD

  - id: wireframe-palette-discipline
    rule: "..."
    severity: required
    check_method: mechanical # ADD
```

Map: the 7 design principles are mostly `semantic`; the 6 mechanical rules are `mechanical`. Both methods can be `both` for the few that have mechanical pre-checks plus semantic depth.

### Optional

**Add `selection_signal: surface_fill` to `components.list_item.variants[id=selected]`** (Change 10).

**Add structured `anatomy` with `yields_to` to `components.list_item`** (Change 11). The wireframe library's list_item already enforces this in code; promoting the constraint to the schema is documentation-as-rule.

### Scope
Involves adding `check_method` annotations to all 13 rulebook entries. Optional updates to list_item involve adding `selection_signal` and structured `anatomy` declarations.

## Terminal

### Required

**Add `requires_unicode_blocks` to `tokens.typography.families.structural`** (Change 6):

```yaml
typography:
  families:
    - id: structural
      stack: "'JetBrains Mono', ui-monospace, monospace"
      role: every piece of text in the system
      mixing_rule: "monospace only; no proportional fonts permitted anywhere"
      requires_unicode_blocks:
        - { name: "Box Drawing", range: "U+2500-U+257F", reason: "session frames ┌─┐│└─┘" }
        - { name: "Block Elements", range: "U+2580-U+259F", reason: "cursor block █" }
      fallback_strategy: substitute   # ASCII fallback for missing glyphs
```

**Add `style: dashed` to the dashed border treatment** (Change 5). Terminal's `elevation.config.border_light` should declare `style: dashed`:

```yaml
elevation:
  strategy: borders_only
  config:
    border_light:
      width: 1
      color: "#1F2F1F"
      style: dashed
      dash_pattern: "3,2"
    border_strong:
      width: 1
      color: "#1F2F1F"
      style: solid
```

**Add `check_method` to every rulebook entry.** Terminal has 8 rules; most are `mechanical` (monospace-only, no-radius, no-shadows, color-as-signal, snake-case-names), some are `semantic` (prompt-style-labels, box-drawing-borders, selection-via-character).

### Optional

**Add `selection_signal: prompt_character`** to list_item.selected with detail `{ glyph: ">", color_ref: tokens.colors.accent }`.

### Scope
Required: add `requires_unicode_blocks` declarations (requires careful enumeration of glyphs used), add `style: dashed` to border treatment, add `check_method` to all 8 rulebook entries. Optional: add `selection_signal: prompt_character` to list_item.selected.

## Migration Workflow

All 6 systems need structured field additions per the v0.2 schema. None of it is creative work — it's declarative schema compliance. Per-system priority if doing them sequentially:

1. **Riso** — already done (riso-spec-v0.2.yaml). The full demonstration of v0.2.
2. **Terminal** — most distinctive required migration (unicode blocks). Demonstrates Change 6.
3. **Editorial** — first emoji-using system to migrate. Demonstrates Change 7.
4. **Prism** — first format_fitness migration. Demonstrates Change 8.
5. **Swiss / Sketch / Revolt** — straightforward check_method + emoji additions.
6. **Wireframe library** — last, since it's the most stable and the changes are mostly bookkeeping.

The remaining migrations follow the same pattern and can be applied directly without further architectural decisions.

## What v0.2 Doesn't Force You To Do

Worth naming explicitly:

**You don't have to migrate everything.** v0.1 specs work indefinitely under v0.2. The migrations are recommendations for systems that benefit from new features.

**You don't have to use new blocks if you don't need them.** A system without parametric state has no `system_state` block. A system without filters has no `filter_library`. A system without artifact-level treatments has no `artifact_treatments`.

**You don't have to annotate `check_method` immediately.** The default for unannotated rules is `mechanical` (matching v0.1's rulebook_check.py behavior). Semantic rules will simply skip the deeper check until annotated.

The v0.2 features are progressive enhancement — you opt in when the value is clear.
